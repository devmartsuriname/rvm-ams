### Item 2 — Remove Lovable Platform Dependency

**Audit finding:** C4 — `lovable-tagger` top-level import breaks VPS production build
**Files changed:** `vite.config.ts`, `package.json`

**vite.config.ts before:**
```typescript
import { componentTagger } from "lovable-tagger";
...
plugins: [react(), mode === "development" && componentTagger()].filter(Boolean),
```

**vite.config.ts after:**
```typescript
plugins: [react()],
```
(Import removed entirely. `mode` parameter removed from `defineConfig` since no longer needed.)

**package.json:** `"lovable-tagger": "^1.1.11"` removed from devDependencies.

**Why:** The top-level `import { componentTagger } from "lovable-tagger"` executes unconditionally at Vite config parse time. Node resolves imports regardless of whether the imported symbol is called at runtime. On a VPS production build with `npm ci --omit=dev`, this import fails with `Cannot find module 'lovable-tagger'` before any build step runs. `componentTagger` is a Lovable-editor-specific tool with no function in VPS deployments.

### Item 3 — Document Creation: Upload-First Atomicity Fix

**Audit finding:** C2 — Non-atomic document creation; orphaned DB record risk
**File changed:** `src/services/documentService.ts` — `createDocument()` method

**Original operation order:**
```
1. INSERT rvm_document  ← if upload fails at step 2, this record is orphaned forever
2. upload to storage
3. INSERT rvm_document_version
4. UPDATE rvm_document (link version)
```

**New operation order:**
```
1. crypto.randomUUID()  ← pre-generate document ID client-side
2. upload to storage/{docId}/v1/{filename}  ← if this fails: nothing in DB, clean failure
3. INSERT rvm_document with explicit id: docId  ← if fails: remove storage file
4. INSERT rvm_document_version  ← if fails: remove storage file
5. UPDATE rvm_document (link version)
```

**Key technique:** By pre-generating the document UUID client-side, the final storage path can be computed before the DB insert. This enables upload-first ordering without needing a temp path or a copy operation.

**Storage cleanup on failure:**
```typescript
// Step 3 failure:
await supabase.storage.from('rvm-documents').remove([storagePath])

// Step 4 failure:
await supabase.storage.from('rvm-documents').remove([storagePath])
```

**What this fixes:**
- Most common failure (storage/network error): no DB record created ✅
- DB insert failure after upload: storage file cleaned up ✅
- Version insert failure: storage file cleaned up ✅

**Remaining accepted risk:** If the `current_version_id` link (step 5) fails, a `rvm_document` and `rvm_document_version` exist but are not linked. This is recoverable (admin can manually link, or re-upload creates a new version). This scenario requires simultaneous DB success at steps 3–4 and failure at step 5, which is rare.

### Item 4 — Dossier Creation: Surface Orphan ID in Error

**Audit finding:** C1 — Non-atomic dossier creation; misleading error on failure
**File changed:** `src/services/dossierService.ts` — `createDossier()` error handler

**Before:**
```typescript
if (itemError) {
  // Rollback dossier if item creation fails  ← WRONG COMMENT
  console.error('[DossierService] Failed to create item, dossier orphaned:', itemError)
  throw itemError  // raw Supabase error, no orphan ID surfaced
}
```

**After:**
```typescript
if (itemError) {
  console.error('[DossierService] rvm_item creation failed. Orphaned dossier ID:', dossier.id, itemError)
  throw new Error(
    `Dossier was created but the supplementary record failed to save. ` +
    `Orphaned dossier ID: ${dossier.id}. ` +
    `Please contact your administrator to resolve this incomplete record.`
  )
}
```

**Why:** The original code had a comment saying "rollback dossier" but zero rollback logic. The orphaned dossier was invisible — neither the error message nor logs identified which record needed cleanup. The new error surfaces the exact UUID so an administrator can find and resolve the incomplete record via Supabase dashboard or SQL.

**Why not fully fixed:** True atomicity for `rvm_dossier` + `rvm_item` creation requires a database-level stored procedure (an RPC function wrapping both inserts in a transaction). This is a schema change, which is out of scope for Phase 26A. The mitigation (surfaced ID) makes the failure recoverable rather than silently corrupting data.

## 4. Risks Introduced

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Missing `.env` at build time now causes explicit startup failure | **Intentional** | The fail-fast error message explains exactly what is needed |
| `crypto.randomUUID()` availability | **Very Low** | `crypto.randomUUID()` is available in all modern browsers and Node.js 15+. Not a concern for Vite/React builds. |
| Pre-generated UUID client-side could theoretically collide | **Negligible** | UUID v4 collision probability is astronomically small. PostgreSQL `gen_random_uuid()` uses the same algorithm. |
| Removing `lovable-tagger` breaks Lovable editor component tagging | **Low / Intentional** | The Lovable editor may lose component selection features. This is an acceptable trade-off for VPS build compatibility. Lovable can be used without tagging. |
